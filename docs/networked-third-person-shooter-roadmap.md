# 第三人称联网射击游戏递进开发任务书

本文档用于指导当前 Unity Tank 原型，逐步改造成一个适合实习作品集展示、能讲清楚实现原理、代码不故意炫技的第三人称联网射击项目。

项目目标不是做一套真正商业级网络框架，而是做出一个“新手能吃透、面试能讲清、效果能展示”的联网射击 Demo。

项目开发过程本身也是学习过程。前期允许使用更容易理解、更容易调试的实现方式；当你掌握核心概念后，再逐步替换和优化之前的实现。每次引入新技术之前，都要先讲清楚：

- 为什么当前阶段需要它。
- 它解决了什么具体问题。
- 有哪些常见替代方案。
- 为什么现在先选这个方案。
- 未来什么时候可以替换成更专业的方案。

当前项目起点：

- Unity 客户端已有 `Player`：本地输入、坦克移动、炮塔瞄准、子弹发射。
- Unity 客户端已有 `Target`：本地碰撞命中反馈。
- `Server/Server` 已存在 .NET 8 控制台项目，但目前只是模板程序。

## 项目定位

这个项目的最终展示目标：

- 能启动一个本地服务器。
- 能打开两个 Unity 客户端进入同一场景。
- 两个玩家能看到彼此移动、转向、瞄准、开火。
- 服务器负责计算关键状态，例如位置、血量、死亡。
- 客户端有本地预测，所以自己操作不明显卡顿。
- 客户端有插值和平滑修正，所以别人移动不明显抖动。
- 能模拟 100 ms 到 200 ms 延迟，并说明为什么仍然可玩。
- 能在面试中清楚解释：Tick、输入包、状态快照、预测、回滚、插值、服务器权威。

不作为当前阶段目标：

- 不做复杂匹配系统。
- 不做账号登录系统。
- 不做大型房间管理系统。
- 不做复杂反作弊系统。
- 不做云服务器部署。
- 不做完整商业级武器、技能、物理弹道系统。
- 不追求极致性能优化。

## 代码风格规定

为了保证你能理解并讲清楚，本项目代码要遵守以下原则：

1. 优先写直白、可读、可调试的代码。
   - 宁可多写几行清楚的 `if` 和 `switch`，也不要一开始写复杂框架。
   - 避免为了“高级”而使用难懂的泛型、反射、委托链、复杂回调。

2. 少用隐藏控制流。
   - 网络消息处理第一版使用清晰的 `switch (messageType)`。
   - 反射分发可以作为后续优化或对比点，不作为第一版必需内容。

3. 每个脚本只做一件主要事情。
   - 输入脚本只读输入。
   - 移动脚本只移动。
   - 网络脚本只收发消息。
   - 表现脚本只做画面平滑和特效。

4. 所有核心概念都要能画图解释。
   - 客户端每 tick 发送输入。
   - 服务器每 tick 计算世界。
   - 服务器广播状态快照。
   - 客户端预测自己。
   - 客户端插值别人。
   - 收到服务器状态后对自己做回滚修正。

5. JSON 序列化优先。
   - 第一版网络消息使用 JSON，方便打印日志和调试。
   - 后续如果需要优化，再考虑二进制序列化。

6. 先理解，再升级。
   - 第一版代码优先做到直观、可运行、可讲解。
   - 后续可以在同一个功能点上做第二版、第三版优化。
   - 优化不能只为了看起来高级，必须能说明它解决了什么问题。

7. 每个技术选择都要有解释。
   - 使用 UDP 前，要讲清楚 UDP 和 TCP 的区别。
   - 使用 Tick 前，要讲清楚为什么不用每帧直接同步 Transform。
   - 使用状态快照前，要讲清楚快照和事件消息的区别。
   - 使用客户端预测前，要讲清楚为什么只靠服务器同步会手感差。
   - 使用回滚前，要讲清楚为什么预测会出错，以及为什么要重放输入。

## 总体技术方案

1. 通信方式：UDP + JSON。
   - UDP 用来练习实时游戏网络同步。
   - JSON 用来降低学习成本，方便在控制台直接看消息内容。
   - JSON 确实不是高性能实时 FPS 的最终选择，但它非常适合作为学习版协议。
   - 未来可以替换为 MessagePack、MemoryPack、Protocol Buffers 或自定义二进制协议。

2. 网络模型：服务器权威为主。
   - 客户端发送输入。
   - 服务器根据输入计算玩家位置、血量、命中结果。
   - 客户端不能直接决定最终血量和最终位置。

3. 同步方式：Tick + 状态快照。
   - 服务器固定 Tick，例如 30 次每秒。
   - 客户端按固定频率发送输入。
   - 服务器按固定频率广播快照。

4. 开火和命中：先做简单实时事件。
   - 客户端发送开火请求。
   - 服务器校验后立即转发开火事件。
   - 命中先做服务端射线检测，不做复杂弹道。

5. 延迟处理：预测、插值、简单回滚。
   - 本地玩家使用客户端预测，保证操作跟手。
   - 远端玩家使用快照插值，保证移动平滑。
   - 本地玩家收到服务器权威状态后，必要时回滚并重放未确认输入。

## 技术选型学习路线

本项目不要求一次选到最强方案，而是按照“先看懂，再优化”的顺序推进。

### UDP 与 TCP

当前选择：UDP。

为什么用：

- 实时游戏更关心“最新状态”，而不是保证每个旧状态都必须到达。
- 玩家位置、旋转、瞄准方向这类数据会不断更新，旧包晚到时价值很低。
- UDP 更适合练习延迟、丢包、乱序、快照同步这些核心概念。

可替代方案：

- TCP：可靠、有序，适合登录、聊天、背包、商城等数据，但实时移动同步容易被旧包阻塞。
- WebSocket：上手简单，适合网页游戏或工具，但底层通常基于 TCP。
- KCP：基于 UDP 做可靠传输，适合需要可靠又想控制延迟的场景。
- LiteNetLib：C# UDP 网络库，封装了连接、可靠消息、NAT 等功能。

当前为什么不用更复杂方案：

- 你现在最重要的是理解“游戏同步到底在同步什么”。
- 直接上复杂网络库会隐藏很多细节，不利于学习。
- 第一版用 .NET 自带 `UdpClient` 更容易看懂每一步。

未来升级方向：

- 当 `UdpClient + JSON` 跑通后，可以把传输层替换成 LiteNetLib。
- 替换时保持上层消息结构不变，只改底层收发。

### JSON 与 Protocol Buffers

当前选择：JSON。

为什么用：

- 可读性强，控制台打印出来就是人能看懂的文本。
- Unity 和 .NET 都容易处理。
- 前期调试网络消息时，JSON 能直接帮助你理解客户端和服务器在说什么。

JSON 的问题：

- 体积比二进制协议大。
- 序列化和反序列化速度较慢。
- 字段名会重复发送，浪费带宽。
- 对高频实时同步来说，不是最终性能方案。

可替代方案：

- Protocol Buffers：成熟、跨语言、体积小，常见于网络服务和部分游戏项目。
- MessagePack：二进制格式，C# 使用方便，体积和速度都比 JSON 好。
- MemoryPack：C# 场景下性能很强，但学习成本略高。
- 自定义二进制协议：性能和体积可控，但最容易写错，也最难调试。

当前为什么先不用 Protocol Buffers：

- 第一阶段重点是理解 Tick、快照、预测和回滚。
- Protocol Buffers 需要额外定义 `.proto` 文件和生成代码，会增加学习负担。
- JSON 慢的问题在当前小规模 Demo 中不会成为主要瓶颈。

未来升级方向：

- 阶段 1 到阶段 7 完成后，可以做一次“协议层升级任务”。
- 目标是把 JSON 消息替换成 MessagePack 或 Protocol Buffers。
- 替换前后对比包体大小、发送频率、CPU 开销和代码复杂度。

### Switch 消息分发与反射消息分发

当前选择：`switch`。

为什么用：

- 新手最容易读懂。
- 调试时能清楚看到每种消息走哪个分支。
- 编译器能帮你检查一些明显错误。

可替代方案：

- 字典分发：`Dictionary<string, Action<Message>>`。
- 观察者模式：多个系统订阅自己关心的消息。
- 反射分发：根据消息类型自动调用对应处理函数。

当前为什么不先用反射：

- 反射会隐藏调用关系。
- 出错时不如 `switch` 好查。
- 对当前项目来说，消息类型数量不会特别多。

未来升级方向：

- 当消息类型变多后，可以先升级成字典分发。
- 如果你已经理解消息流，再尝试反射分发作为架构亮点。

### 服务器权威与 P2P

当前选择：服务器权威。

为什么用：

- 面试中这是多人竞技游戏最重要的基础之一。
- 服务器决定位置、血量和命中，客户端作弊空间更小。
- 两个客户端发生冲突时，以服务器结果为准。

可替代方案：

- P2P：玩家之间直接同步，延迟可能更低，但反作弊和一致性更难。
- 主机模式：一个玩家同时当客户端和服务器，适合小型合作游戏。
- 完全客户端权威：实现简单，但竞技游戏很容易作弊。

当前为什么选服务器权威：

- 你的目标是第三人称联网射击，服务器权威最适合讲解公平性和同步一致性。
- 即使 Demo 很小，这个方向也是正确的。

## 逐步优化原则

后续开发允许这样演进：

1. 第一版：写最简单、最直观的实现，保证你能跑通和讲清楚。
2. 第二版：根据问题做局部优化，例如减少消息体积、减少重复代码、增加调试面板。
3. 第三版：在你理解之后替换底层方案，例如从 JSON 换到 MessagePack 或 Protocol Buffers。
4. 每次升级都要保留对比说明：旧方案的问题是什么，新方案解决了什么，新方案带来了什么复杂度。

这意味着项目最终可以达到甚至超过同学项目的展示程度，但过程不是照抄他的技术词，而是把每个词背后的原理真正做出来。

## 阶段 0：整理当前单机 Tank

目标：把当前 `Player` 脚本拆成更容易接网络的结构。

任务：

- 新建 `TankInput`。
  - 读取键盘和鼠标输入。
  - 输出一个简单的输入数据结构。
- 新建 `TankMotor`。
  - 根据输入移动和旋转坦克。
  - 暂时保持现有手感。
- 新建 `TankAim`。
  - 负责鼠标射线瞄准。
  - 负责炮塔朝向目标点。
- 新建 `TankWeapon`。
  - 负责开火入口。
  - 第一版仍然可以本地生成子弹。
- 保留一个 `TankController` 做组合。
  - 把输入、移动、瞄准、武器串起来。

验收标准：

- 单机移动、瞄准、射击仍然正常。
- 每个脚本职责清楚。
- 后面可以很自然地把输入数据发给服务器。

当前进度：

- 状态：阶段 0 脚本拆分已完成，等待在 Unity 编辑器中完成手动挂载和 Play 验收。
- 已新增 `TankInputData`：作为输入数据包，保存移动、转向、开火和鼠标位置。
- 已新增 `TankInput`：只读取键盘和鼠标输入，并输出 `TankInputData`。
- 已新增 `TankMotor`：只根据输入移动坦克和旋转车身。
- 已新增 `TankAim`：只负责鼠标射线瞄准和炮塔朝向目标点。
- 已新增 `TankWeapon`：只负责本地开火入口和生成子弹。
- 已新增 `TankController`：只负责组合输入、移动、瞄准和武器模块。
- 当前协作方式：Codex 只编写脚本，不直接修改 Unity 场景、Prefab、组件挂载和对象引用；组件挂载和引用拖拽由用户在 Unity 编辑器中完成。

## 阶段 1：最小 UDP 服务器

目标：让 `Server/Server` 从模板程序变成能接收 Unity 客户端消息的服务器。

任务：

- 使用 .NET 自带的 `UdpClient`。
- 服务器启动后监听一个端口，例如 `7777`。
- Unity 客户端发送 `ClientHello`。
- 服务器回复 `ServerWelcome`，分配 `playerId`。
- 控制台打印连接日志和收到的 JSON 消息。

第一版消息结构示例：

```json
{
  "type": "ClientHello",
  "name": "Player"
}
```

验收标准：

- 服务器能启动。
- Unity 客户端能连接。
- 服务器能给客户端分配玩家 id。
- 所有消息都能在控制台看懂。

## 阶段 2：Tick 输入与服务器权威移动

目标：让服务器开始接管玩家位置计算。

任务：

- 客户端每个网络 tick 发送输入。
- 输入包包含：
  - `playerId`
  - `inputTick`
  - `moveAxis`
  - `turnAxis`
  - `aimX`
  - `aimZ`
  - `fire`
- 服务器保存每个玩家的最新输入。
- 服务器以 30 Hz Tick 更新玩家位置。
- 服务器广播 `WorldSnapshot`。
- 客户端根据快照更新玩家位置。

阶段 2 基础验收标准：

- 单个 Unity 客户端能连接服务器，并持续发送 `PlayerInput`。
- 服务器能根据客户端输入更新玩家权威位置。
- 服务器能持续广播 `WorldSnapshot`。
- 客户端能根据服务器快照更新本地玩家位置。
- 服务器控制台能看到 tick、输入数量和快照数量增长。
- 服务器决定玩家最终位置。
- 客户端不能单靠本地改速度来改变服务器权威位置。

说明：

- 两个客户端互相可见、远端坦克创建、延迟模拟软件测试，单独放到阶段 2.5。
- 这样做的原因是：阶段 2 先确认“输入 -> 服务器 Tick -> 快照”的核心链路正确；阶段 2.5 再确认这条链路在两个客户端和模拟延迟下也能观察、讲清楚。

## 阶段 2.5：双客户端联调与延迟模拟准备

目标：在进入远端插值前，先把“两个客户端互相看见”和“模拟网络延迟”的测试环境搭好。

为什么需要这个阶段：

- 阶段 2 已经跑通服务器权威移动的基础代码，但目前主要验证的是单个客户端和服务器之间的通信。
- 阶段 3 要解决“别人移动一跳一跳”的问题；在写插值前，必须先能稳定看到“对方坦克”。
- 延迟模拟能提前暴露真实网络环境下的问题，例如快照延迟、抖动、丢包和远端显示卡顿。
- 这个阶段不急着写插值代码，而是先建立可靠的测试方法，避免后面不知道问题来自代码还是测试环境。

任务：

- 创建或整理远端玩家使用的坦克 Prefab。
  - Prefab 内部要有坦克模型、炮塔、炮口和可选的 Aim 子物体。
  - Prefab 上可以保留 `TankController`，但远端实例生成后会禁用本地输入和本地控制。
  - Prefab 上建议挂 `NetworkTankAvatar`，并拖好 Prefab 内部的炮塔引用。
  - 如果 Prefab 内部没有 Aim 子物体，`NetworkTankAvatar.aimTransform` 可以先留空。
- 正确配置场景中的 `UdpNetworkClient`。
  - `localTank` 指向场景中的本地玩家坦克。
  - `localAvatar` 指向本地玩家坦克上的 `NetworkTankAvatar`。
  - `remoteTankPrefab` 指向远端玩家使用的坦克 Prefab。
  - `serverAuthoritativeMovement` 保持开启。
- 使用两个客户端连接同一个本地服务器。
  - 推荐方式：Unity Editor 作为客户端 A，Build 出来的 Windows 客户端作为客户端 B。
  - 两个客户端都连接 `127.0.0.1:7777`。
  - 两个客户端应获得不同的 `playerId`。
- 验证远端玩家创建和同步。
  - 客户端 A 移动时，客户端 B 能看到 A 的坦克位置变化。
  - 客户端 B 移动时，客户端 A 能看到 B 的坦克位置变化。
  - 服务器控制台能看到在线玩家数、输入数量和快照数量持续增长。
- 使用延迟模拟软件做初步测试。
  - 先模拟 100 ms 延迟。
  - 再模拟 200 ms 延迟。
  - 可选：加入少量抖动和丢包。
  - 记录不同延迟下本地玩家和远端玩家的表现差异。

延迟模拟建议：

- Windows 本机可以使用常见网络延迟模拟工具，例如 clumsy 这类 UDP 延迟/丢包模拟工具。
- 第一轮只模拟 UDP 端口 `7777` 的延迟，避免影响系统里其他网络程序。
- 初始参数建议：
  - 100 ms 固定延迟，不加丢包。
  - 200 ms 固定延迟，不加丢包。
  - 100 ms 固定延迟 + 少量 jitter。
  - 100 ms 固定延迟 + 1% 到 3% 丢包。
- 当前阶段只观察现象，不要求“修好”延迟带来的卡顿；远端卡顿主要留给阶段 3 插值解决，本地手感延迟主要留给阶段 4 预测解决。

验收标准：

- 能启动一个本地服务器。
- 能同时打开两个客户端进入同一场景。
- 两个客户端能看到彼此的坦克。
- 两个客户端移动时，服务器都会更新对应玩家的权威位置。
- 两个客户端都能收到包含两个玩家的 `WorldSnapshot`。
- 100 ms 延迟下，能观察到远端移动延后或跳动。
- 能说清楚：当前阶段为什么会跳动，以及下一阶段为什么要做远端插值。

本阶段不做：

- 不做远端玩家插值。
- 不做客户端本地预测。
- 不做回滚修正。
- 不做服务器权威开火和命中。
- 不把延迟模拟写进游戏代码；先使用外部软件观察网络现象。

## 阶段 3：远端玩家插值

目标：让“别人”的移动平滑。

任务：

- 客户端为远端玩家保存最近几帧快照。
- 渲染时不直接使用最新快照，而是稍微落后一点播放。
- 在两个快照之间做 `Vector3.Lerp` 和 `Quaternion.Slerp`。
- 如果短时间没收到新快照，可以保持最后状态或短暂外推。

验收标准：

- 远端玩家不再明显一跳一跳。
- 在 100 ms 延迟下，远端移动仍然能接受。
- 插值代码能用简单图解释清楚。

## 阶段 4：客户端本地预测

目标：让“自己”的操作不卡手。

任务：

- 本地玩家按键后，客户端立即先移动自己。
- 同时把这份输入发送给服务器。
- 客户端保存最近一段输入历史。
- 服务器快照中带上 `lastProcessedInputTick`。

验收标准：

- 即使有 100 ms 延迟，自己按下移动键时也能立刻动。
- 服务器仍然会返回权威位置。
- 客户端知道服务器已经处理到哪一帧输入。

## 阶段 5：简单回滚与平滑修正

目标：让客户端预测和服务器权威状态不一致时，可以修回来。

任务：

- 收到服务器快照后，对比本地预测位置和服务器位置。
- 如果误差很小，使用平滑追赶。
- 如果误差超过阈值，回到服务器位置。
- 回到服务器位置后，重放还没有被服务器确认的输入。
- 回滚阈值根据 RTT 做简单调整。
  - RTT 低时阈值小一点。
  - RTT 高时阈值大一点。

验收标准：

- 位置不会长期漂移。
- 轻微误差不会突然瞬移。
- 大误差能被及时修正。
- 能解释“为什么要回滚重放输入”。

## 阶段 6：联网开火与命中

目标：把开火和命中从本地行为改成网络事件。

任务：

- 客户端按下鼠标左键后发送 `FireRequest`。
- 服务器检查玩家是否可以开火。
  - 是否存活。
  - 射击间隔是否足够。
  - 炮塔方向是否合理。
- 服务器做一次简单射线检测。
- 服务器广播：
  - `FireEvent`
  - `HitEvent`
  - `HealthChangedEvent`
- 客户端收到事件后播放开火、命中和掉血表现。

验收标准：

- 两个客户端都能看到开火。
- 命中和扣血由服务器决定。
- 客户端不能自己说“我命中了，所以你扣血”。

## 阶段 7：血量、死亡、重生

目标：形成一个完整的战斗闭环。

任务：

- 服务器维护玩家血量。
- 玩家被命中后扣血。
- 血量归零后进入死亡状态。
- 几秒后重生。
- 客户端显示简单血量和死亡状态。

验收标准：

- 玩家能互相攻击。
- 玩家会死亡和重生。
- 两个客户端看到的血量一致。

验收标准：

阶段 5 讲解要点：

- 只做客户端预测会有一个问题：客户端先猜自己在哪里，服务器稍后返回权威结果，如果两边计算规则、输入到达时间或网络延迟不完全一致，本地位置就可能慢慢漂。
- 回滚的做法不是直接相信当前客户端位置，而是回到服务器已经确认的那一帧权威状态。
- 重放的原因是：服务器快照只确认到 `lastProcessedInputTick`，但客户端手里可能已经输入了更新的几帧。如果只回到服务器位置，不重放这些未确认输入，玩家会感觉自己被拉回过去。
- 所以正确流程是“服务器权威状态 -> 重放未确认输入 -> 得到当前应该显示的预测状态”。
- 小误差平滑追赶，是为了避免轻微数值差异造成肉眼可见瞬移。
- 大误差立即修正，是为了防止位置长期漂移，或者客户端预测已经明显跑偏。
- RTT 高时阈值适当放宽，因为高延迟下未确认输入更多，客户端和服务器短时间差异也更自然；RTT 低时阈值更小，可以更快修回。
